## Suite de prompts backend ILMA (FastAPI 3.11+)

### PROMPT 00 — “Règles globales” (à coller en premier)

**Objectif :** imposer les standards d’architecture, la qualité, et éviter une base “démo”.

Tu es un ingénieur backend senior. Génère un backend FastAPI async (Python 3.11+) pour ILMA (application éducative CM2).
Respecte une architecture “clean-ish”: route -service -repository -db. 

STACK OBLIGATOIRE: FastAPI async, Pydantic v2, SQLAlchemy 2.0 async, Alembic, PostgreSQL 15+. Redis optionnel (cache + rate limit). Auth JWT access+refresh. Docker compose (api + postgres + redis). Tests pytest async. 

API: base path `/api/v1`, pagination/filtre/tri, OpenAPI propre + tags. Format réponse standard `{success,data,message,timestamp}` et erreurs `{success:false,error:{code,message,details},timestamp}`. 
Ajoute `GET /health` et `GET /version`. 

OUTPUT ATTENDU À CHAQUE LIVRAISON:

1. Tree projet + brève explication
2. Code complet prêt à build
3. Migrations Alembic
4. Docker compose + `.env.example` + seed script
5. Tests minimaux par module 

Contraintes ILMA clés: offline-first + sync opportuniste + gestion de conflits + idempotence sur événements offline (ex: attempts avec `client_event_id` unique).  

Ne laisse pas de TODO bloquants : si une intégration externe (SMS, paiement, push) manque, implémente une interface + un provider “Mock” + un provider “HTTP” configurable.

Le projet du frontend est déjà présent dans le répertoire frontend
Le code du backend devra être dans le répertoire backend
DSC_Ilma_Complet_V2.docx et DSFT_Ilma_V3_Complete.docx dans le répertoire courant.

### PROMPT 03 — Auth (OTP + email) + JWT + audit

**Objectif :** endpoints auth DSFT, OTP rules DSC, sécurité.

Implémente l’auth ILMA avec les endpoints: `/auth/register`, `/auth/login`, `/auth/refresh`, `/auth/logout`, `/auth/otp/send`, `/auth/otp/verify`, `/auth/password/reset`, `/auth/password/change`. 
OTP: 6 chiffres, validité 5 minutes, 3 tentatives max, renvoi après 60s, validation format téléphone +229XXXXXXXX + préfixes bénin. 
Hash password bcrypt cost 12 ou Argon2id. Ajoute rate limit (Redis si dispo), journaux d’audit (login success/fail), et tests (ok/ko).

### PROMPT 09 — Offline packs (ZIP) + endpoint sync (cœur production)

**Objectif :** packs + sync + conflits + statut.

Implémente les endpoints offline:

* `GET /offline/packs`
* `GET /offline/packs/{id}/download` (ZIP stream + ETag + version)
* `POST /offline/sync` (upload batch événements offline)
* `GET /offline/sync/status` 

Règles sync à implémenter: priorité upload attempts puis badges puis profil, ensuite download updates. Gestion conflits: meilleur score si même exercice, last-write-wins pour profil, ignorer doublons badge, downgrade si abonnement expiré. 

Le payload `/offline/sync` doit accepter une liste d’événements typés (attempt_created, session_completed, badge_gained, profile_updated) avec `client_event_id` unique pour idempotence. 
Ajoute tests (replay batch, doublons, conflits).

## Pourquoi ce n’est pas “production-ready” sans ces prompts

Pour ILMA, si tu n’as pas explicitement :

* **idempotence** (client_event_id unique) 
* **conflits offline** (meilleur score / last-write-wins / downgrade abonnement) 
* **rate limiting OTP/login** + audit
* **tests sur sync** (replay, doublons)
  alors tu auras des bugs “invisibles” qui explosent en prod.

Si tu veux, je peux ensuite te donner une **checklist DoD backend “production-ready”** alignée sur cette suite (sécurité, perf, tests, observabilité, runbook), mais la base ci-dessus est le minimum pour éviter un backend “démo”.
